# API Synchronization Approach

## Current State

- **Backend**: Django REST Framework with `drf-spectacular` for OpenAPI schema generation.
- **Frontend**: Vite/React application with a hardcoded `API_ENDPOINTS` object in `src/constants/apiEndpoints.js`.

## Challenges

- The frontend's `API_ENDPOINTS` must be kept in sync with the backend's actual API endpoints.
- Manual updates are error-prone and require constant vigilance.

## Proposed Solution

Use the OpenAPI schema (generated by `drf-spectacular`) as the **single source of truth** for the API contract.

### Steps

1. **Backend**: Ensure the `drf-spectacular` settings are correct and the schema is accessible at `/api/schema/`.
2. **Generation**: Use a script (to be run during frontend build or as a precommit hook) to:
   a. Fetch the OpenAPI schema from the backend (or use a local copy if generated during backend build).
   b. Parse the schema to extract all endpoints and their paths.
   c. Generate a JavaScript module that exports an object similar to the current `API_ENDPOINTS`, where:
      - Static endpoints are exported as strings.
      - Endpoints with path parameters are exported as functions that accept the parameters and return the string with the parameters replaced.
3. **Integration**: Replace the current `src/constants/apiEndpoints.js` with the generated file (or generate it and commit it, or generate it at build time).

## Benefits

- Guarantees that the frontend constants are always in sync with the backend API.
- Reduces manual effort and the chance of human error.
- Allows the backend to evolve independently, and the frontend can be updated by regenerating the constants.

## Implementation Notes

- The base URL (from `import.meta.env.VITE_API_BASE_URL`) should be kept separate, as it is environment-dependent.
- The generated file should only contain the endpoint paths (without the base URL) and the frontend should prepend the base URL as it does now.

## Example

Given an OpenAPI path: `/api/articles/{id}/`
We would generate:
```javascript
ARTICLE_DETAIL: (id) => \`/api/articles/\${id}/\`
```

And then in the frontend, when using it:
```javascript
const url = \`\${API_BASE_URL}\${API_ENDPOINTS.ARTICLE_DETAIL(articleId)}\`
```

Note: This requires a slight refactor of the frontend to use the constants in this way (combining with the base URL).

## Migration Plan

1. Create a script to generate the API constants from the OpenAPI schema.
2. Update the frontend services to use the generated constants (by combining with the base URL).
3. Replace the current `apiEndpoints.js` with the generated one (or generate it and commit it, or generate it at build time).
4. Set up a CI step to verify that the generated constants match the backend schema (or generate them on every build).

## Alternative: Runtime Validation

We can also consider keeping the constants as they are and adding a test that fetches the OpenAPI schema and validates that every constant (when combined with the base URL) matches a path in the schema.

However, the generation approach is more robust because it ensures the constants are correct by construction.

## Conclusion

We recommend generating the frontend API constants from the backend's OpenAPI schema.

Let's create the script and update the frontend accordingly.
